# Task Breakdown: Bootstrap the Ascend repository (Issue #3)

All tasks implement **ADR-0002 — Ascend baseline technology stack and repository layout**
and must comply with **CORE-COMPONENT-0002 (Commit Standards)** for commit/PR authoring.
No task may add an application framework, application feature, or any DevDeck code.

Tasks are ordered by dependency. Complexity is relative (S / M / L).

---

## Task T1: Create the minimal project manifest and toolchain

- **Status:** Not started
- **Complexity:** M
- **Dependencies:** None
- **Related ADRs:** ADR-0002
- **Related Core-Components:** CORE-COMPONENT-0002

### Description
Create the minimal manifest and toolchain that materialise the ADR-0002 stack decision:
- `package.json` with project metadata (name `ascend`, private, description, license field),
  an `engines` field pinning Node.js `>=22 <23`, and scripts limited to what the toolchain
  needs (e.g. `typecheck` running `tsc --noEmit`). No application dependencies; `typescript`
  as the only dev dependency.
- `package-lock.json` generated by `npm install` for reproducible installs (committed).
- `tsconfig.json` — minimal, strict TypeScript config with `rootDir: src` and no build
  output required at bootstrap.
- `.nvmrc` pinning the Node.js major version (`22`).

Do **not** add a web framework, bundler, runtime, or any application dependency.

### Acceptance Criteria
- [ ] `package.json` exists at repo root with `name`, `private: true`, `engines.node`
  pinning Node 22, and a `typecheck` script.
- [ ] The only dependency present is `typescript` under `devDependencies`; no runtime or
  framework dependencies exist.
- [ ] `tsconfig.json` exists with `strict: true` and `rootDir` pointing at `src/`.
- [ ] `.nvmrc` exists and specifies Node `22`.
- [ ] `package-lock.json` is committed and consistent with `package.json`.
- [ ] No DevDeck code, config, or naming is copied into any file.

### Test Coverage
- Manifest validation test (see TEST T1-a): `package.json` parses as valid JSON and
  contains the required fields (`engines.node`, `private`, `typecheck` script).
- Dependency-minimality check (TEST T1-b): assert `dependencies` is empty/absent and
  `devDependencies` contains only `typescript`.
- Toolchain smoke test (TEST T2-b): `npm run typecheck` exits 0 against the empty `src/`.

## Task T2: Establish the documented directory layout

- **Status:** Not started
- **Complexity:** S
- **Dependencies:** T1
- **Related ADRs:** ADR-0002
- **Related Core-Components:** CORE-COMPONENT-0002

### Description
Create the top-level directory layout defined in ADR-0002. Establish `src/` as the agreed
home for application source **without** adding application logic — include only a
`.gitkeep` (or equivalent placeholder) so the directory is tracked. Confirm `docs/` and
`project/` remain in place. The layout must exactly match what the README will document.

### Acceptance Criteria
- [ ] `src/` exists and is tracked by git (contains only a placeholder such as `.gitkeep`).
- [ ] No application source, framework scaffolding, or feature code exists under `src/`.
- [ ] The on-disk top-level layout matches the layout table in ADR-0002 and the README.
- [ ] `docs/` and `project/` are unchanged and present.

### Test Coverage
- Layout structure test (TEST T2-a): assert required paths exist (`src/`, `docs/`,
  `project/`, `package.json`, `tsconfig.json`, `.nvmrc`, `README.md`).
- Toolchain smoke test (TEST T2-b): `npm run typecheck` succeeds against the empty `src/`,
  proving the location is wired into `tsconfig.json`.

## Task T3: Rewrite the README with product boundary, layout, and setup entry point

- **Status:** Not started
- **Complexity:** M
- **Dependencies:** T1, T2
- **Related ADRs:** ADR-0002
- **Related Core-Components:** CORE-COMPONENT-0002

### Description
Replace the template placeholder `README.md` with Ascend content:
- A **product boundary statement** making explicit that *Ascend orchestrates cross-project
  workflow (open/close/switch/resume projects) and VS Code / `code-server` provides the
  IDE* (PRD §5.1, §30). It must not imply Ascend builds IDE features.
- A **documented directory structure** matching the ADR-0002 layout and the actual repo.
- A **"Getting Started" section** documenting the single setup entry point: clone, ensure
  the pinned Node.js version (reference `.nvmrc`), then run `npm install` from the repo
  root — with no further manual steps.
- Links to `CONTRIBUTING.md`, `AGENTS.md`, `docs/`, and `project/` as appropriate.

Keep it concise; do not document features that do not yet exist (health endpoint,
code-server launch).

### Acceptance Criteria
- [ ] README no longer contains template placeholder text ("Project Name", generic copy).
- [ ] README contains a clear product-boundary statement (Ascend orchestrates; VS Code /
  code-server provides the IDE).
- [ ] README documents the directory structure and it matches the on-disk layout.
- [ ] README's "Getting Started" documents exactly one setup entry point (`npm install`)
  and the Node.js version requirement.
- [ ] README does not describe not-yet-built application features.

### Test Coverage
- README content test (TEST T3-a): assert the README contains the product-boundary
  statement (mentions both "orchestrates" and "code-server"/"VS Code") and no residual
  placeholder string "Project Name".
- README setup-doc test (TEST T3-b): assert a "Getting Started" section documents
  `npm install` as the single setup command and references the pinned Node version.
- Layout-consistency check (TEST T3-c): every directory named in the README exists on disk.

## Task T4: Verify clean-checkout setup and DevDeck-free baseline

- **Status:** Not started
- **Complexity:** M
- **Dependencies:** T1, T2, T3
- **Related ADRs:** ADR-0002
- **Related Core-Components:** CORE-COMPONENT-0002

### Description
Validate the whole bootstrap end-to-end from a pristine state: a clean checkout (fresh
clone or `git clean`-equivalent working tree with no `node_modules`) followed by exactly
the documented single entry point must succeed with no additional manual steps. Also
confirm the repository contains no migrated DevDeck code, configuration, or naming.
Ensure `node_modules` is git-ignored so the committed baseline stays clean.

### Acceptance Criteria
- [ ] From a clean checkout with only the pinned Node.js available, `npm install` completes
  successfully with no manual steps beyond those documented in the README.
- [ ] After install, `npm run typecheck` exits 0.
- [ ] `node_modules/` is git-ignored and not committed.
- [ ] A repository scan finds no DevDeck code, config files, or DevDeck-specific naming.
- [ ] The documented setup command in the README is the same one used to set up the repo
  (single entry point, no divergence).

### Test Coverage
- Clean-checkout setup test (TEST T4-a, **P0**): in a pristine tree, run the documented
  `npm install`; assert exit 0 and that `node_modules` is populated.
- Post-setup typecheck test (TEST T4-b): `npm run typecheck` exits 0 after install.
- Ignore-hygiene test (TEST T4-c): assert `.gitignore` excludes `node_modules` and that it
  is not tracked.
- DevDeck-absence test (TEST T4-d): assert no tracked file path or content references
  DevDeck.
